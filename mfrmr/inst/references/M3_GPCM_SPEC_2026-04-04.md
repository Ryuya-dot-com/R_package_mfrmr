# M3 GPCM Specification and Seam Audit

Date: 2026-04-04  
Package baseline: `mfrmr` 0.1.5

Companion planning notes:

- `inst/references/M3_TAM_SIRT_MIRT_ECOSYSTEM_AUDIT_2026-04-04.md`
- `inst/references/M3_ECOSYSTEM_COVERAGE_PLAN_2026-04-04.md`

## 1. Purpose

This note fixes the smallest defensible `GPCM` target for `mfrmr`. The first
release has now moved past scaffolding: narrow public `MML` / `JML` fitting,
core summary output, and fixed-calibration posterior scoring are active.
Broader downstream support is still intentionally incomplete.

The aim is not to add a generic slope model everywhere at once. The aim is to
define:

- the narrowest useful public interface,
- the exact model family to be claimed,
- which current helpers can be reused,
- and which current Rasch-family assumptions must be guarded or rewritten.

## 3. Smallest defensible `GPCM` target

The first `GPCM` release should be:

- unidimensional;
- non-Bayesian;
- compatible with the current `MML` and `JML` architecture only where
  explicitly validated;
- ordered-category only;
- many-facet in the location part of the linear predictor;
- with discrimination attached to exactly one designated non-person facet.

Operationally, that means:

- add `model = "GPCM"` to `fit_mfrm()`;
- keep `step_facet`;
- add `slope_facet`, defaulting to `step_facet`;
- and, in the first release, require `slope_facet == step_facet`.

This keeps the initial claim close to the ordinary item-level `GPCM` logic in
TAM and mirt while still fitting the existing many-facet parameterization.

## 4. Proposed model statement

For person `n`, designated slope-facet level `i`, and ordered category `k`,
the first target should use:

`log(P_k / P_{k-1}) = a_i * (eta_nfi - tau_{ik})`

where:

- `a_i > 0` is the discrimination for level `i` of `slope_facet`;
- `eta_nfi` is the current additive many-facet location predictor without the
  step term;
- `tau_{ik}` are the ordered step parameters for level `i` of `step_facet`.

In the first release:

- the same facet supplies both `a_i` and `tau_{ik}`;
- the rest of the many-facet structure remains additive in location;
- fixing all `a_i = 1` should reduce the model to the current `PCM`.

This reduction is essential. If that reduction is not exact, the first `GPCM`
branch is not ready.

## 5. Proposed public API

### 5.1 Minimal additions

Extend `fit_mfrm()` with:

- `model = c("RSM", "PCM", "GPCM")`
- `slope_facet = NULL`

Rules:

- when `model = "GPCM"`, `step_facet` is required;
- when `slope_facet = NULL`, set `slope_facet <- step_facet`;
- first release requires `identical(step_facet, slope_facet)`.
- the current scaffold already enforces these role rules before any fitting is
  attempted.

### 5.2 First-release exclusions

Do not claim support yet for:

- multi-slope designs across different facets;
- generalized rating-scale parameterizations;
- multidimensional `GPCM`;
- nominal models;
- arbitrary design-matrix slope constraints.

## 6. Current code seams that can be reused

The current codebase already has reusable structure:

1. `fit_mfrm()` and the config pipeline already branch on `model`.
2. `build_indices()` already provides person, facet, and step-facet indexing.
3. The current `MML` and latent-regression quadrature path is model-agnostic
   at the quadrature layer.
4. Category probabilities are already centralized in helper functions
   (`category_prob_rsm()`, `category_prob_pcm()`), which suggests a clean
   `category_prob_gpcm()` insertion point.
5. `compute_information()` already uses a probability-engine helper, which can
   be generalized rather than rewritten from scratch.

## 7. Current blockers and Rasch-only assumptions

The main blockers are structural, not conceptual.

### 7.1 Parameter vector assumptions

Current parameter sizing and expansion are built around:

- additive facet locations,
- and Rasch-family step parameters with fixed discrimination 1.

`GPCM` needs:

- one positive discrimination parameter per `slope_facet` level;
- centering / identification rules that do not fight the discrimination scale;
- and an exact reduction to `PCM` when slopes are fixed at 1.

Current scaffold decision:

- discrimination is parameterized on the log scale;
- `log(a_i)` values are constrained to sum to zero;
- therefore `a_i > 0` and the geometric mean discrimination is 1.

This is a package-specific identification convention. It is mathematically
defensible, but it should not be described as if Muraki (1992), TAM, or mirt
require this exact convention.

### 7.1.1 Additional identification note for future latent regression

The package already has an active latent-regression branch with an estimated
population residual variance under `MML`. Before `GPCM` is allowed to use that
branch, the scale relationship between:

- free discriminations; and
- free latent variance parameters

must be fixed explicitly. The first safe implementation may need to restrict
`GPCM` to a standardized latent scale before allowing broader population-model
combinations.

### 7.2 Probability helpers

Many functions currently switch only between:

- `category_prob_rsm()`
- `category_prob_pcm()`

This affects:

- log-likelihood,
- gradients,
- posterior scoring,
- expected-category summaries,
- bias / DFF machinery that calls response probabilities,
- and information calculations.

### 7.3 Simulation and planning assumptions

`simulate_mfrm_data()` and related planning helpers currently describe data
generation in terms of threshold tables only. A first `GPCM` release will
also need:

- slope generation for `slope_facet` levels;
- a clear simulation contract for the relationship between slopes and
  thresholds;
- and guardrails on which planning helpers are valid before full support is
  implemented.

### 7.4 Reporting semantics

Several current tables and help pages talk as if step parameters are always
Rasch-style thresholds on a common discrimination scale. That wording will
need to split into:

- Rasch-family step interpretation;
- and `GPCM` step-plus-discrimination interpretation.

## 8. First-release downstream boundary

The first `GPCM` release should support only the downstream helpers that can be
validated quickly and coherently.

### 8.1 Must support

- fitting;
- summary;
- posterior scoring under validated methods;
- information helper, if generalized explicitly;
- simulation, if the new generator contract is implemented explicitly.

### 8.2 May temporarily reject with explicit messages

- helpers whose logic is still Rasch-only;
- any automatic nesting logic in `compare_mfrm()`;
- planning summaries that would otherwise imply unsupported inference.

In other words, explicit rejection is better than silent pseudo-support.

## 9. Validation requirements

No `GPCM` implementation should be called complete without:

1. reduction check:
   fixing slopes at 1 reproduces `PCM`;
2. synthetic recovery:
   slopes and steps are recovered in narrow simulated cases; completed for a
   package-native item-only `MML` benchmark case with criterion recovery
   tracked alongside slope and step recovery;
3. scoring check:
   posterior summaries behave sensibly under known simulations;
4. information check:
   the generalized probability engine yields coherent design-weighted curves;
5. documentation:
   help pages explain what differs from `PCM`.

## 10. Immediate implementation queue

The smallest safe queue is:

1. add inactive `GPCM` API scaffolding and guardrails; completed
2. add parameter-size / config scaffolding for `slope_facet`; completed
3. add `category_prob_gpcm()` and reduction tests against `PCM`; completed
4. wire the new probability engine into `MML` and `JML`; completed
5. extend scoring; completed for fixed-calibration posterior scoring
6. extend information; completed for the design-weighted precision screen
7. extend direct simulation; completed for slope-aware simulation
   specifications and `simulate_mfrm_data()`
8. add narrow synthetic recovery and benchmark support; completed via
   `test-parameter-recovery.R` and
   `reference_case_benchmark(cases = "synthetic_gpcm", model = "GPCM")`
9. then widen reporting and planning support only where validated.

## 11. Immediate conclusion

The current first-release `GPCM` target for `mfrmr` remains intentionally
narrow:

> unidimensional many-facet location structure plus one designated slope facet
> that initially matches `step_facet`, with exact reduction to `PCM` when all
> slopes equal 1.

That target now has a package-native recovery benchmark for the currently
validated `MML` branch, but broader reporting and planning generalization
remain outside the completed scope.

That is the smallest extension that is both ecosystem-relevant and
architecturally coherent. As of the current development state, this narrow
target includes fitting, slope summaries, fixed-calibration scoring, and the
design-weighted precision screen from `compute_information()`, plus direct
slope-aware simulation via `build_mfrm_sim_spec()`,
`extract_mfrm_sim_spec()`, and `simulate_mfrm_data()`, but not yet
generalized diagnostics, planning/forecasting, or broad reporting parity.
